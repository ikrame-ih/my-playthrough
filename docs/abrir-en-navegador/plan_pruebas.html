1. Finalidad y alcance

Este documento describe qué se ha probado, cómo y con qué resultado en la aplicación MyPlaythrough. Incluye pruebas manuales (interfaz y API con Thunder Client) y unitarias automáticas (Vitest), sin sustituir la revisión funcional completa del código en el repositorio.

2. Tipos de prueba realizadas

Tipo Descripción breve
Manuales funcionales Comprobación de flujos de usuario (registro, colección, comunidad, admin, comentarios).
Seguridad básica Acceso a rutas protegidas sin token, permisos, límites de cuerpo, límite de frecuencia en login/registro, endpoint de diagnóstico en producción.
Validación de datos Normalización de entradas (email, título, estado, plataforma) coherente con el backend.
Unitarias automáticas Tests repetibles sobre lógica pura (sin navegador ni base de datos), ejecutados con Vitest 3.

Se ejecutan en terminal, desde la carpeta de cada paquete, sin levantar obligatoriamente Docker ni PostgreSQL para los tests del normalize y utilidades del cliente.

Modo vigilancia (opcional): npm run test:watch en server o client. Herramienta: Vitest 3.

9. Cosas que dejaría para más adelante (no están hechas)

  1. Mensajes de error más uniformes: ahora mismo según la ruta el JSON de error puede cambiar un poco de forma. Lo suyo sería pasar todo por el mismo sitio (por ejemplo un middleware) y que en producción el usuario no reciba detalles técnicos largos (stack, cosas internas).
  2. Campana sin ir preguntando cada X segundos: hoy la campana hace polling (cada poco pregunta al servidor si hay novedades). Para un producto grande habría que valorar WebSockets o SSE para que el servidor avise al momento; es más trabajo de despliegue y de reconexión si se cae la red.
  3. Juntar dos fichas del mismo juego: si alguien tiene dos entradas que en realidad son el mismo título, haría falta una pantalla o proceso de administración que las una y mueva comentarios y votos a la ficha buena (sobre todo si ya comparten catalogo_id).
  4. Tests que peguen contra la API de verdad: las unitarias prueban funciones sueltas; faltaría una batería que levante la API (y una base de prueba) y compruebe rutas enteras, tipo Supertest, para pillar fallos de integración entre capas.
