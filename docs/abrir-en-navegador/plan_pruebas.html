1. Purpose and scope

This document describes what was tested, how and with what result in the MyPlaythrough application. It includes manual tests (interface and API with Thunder Client) and automated unit tests (Vitest), without replacing a full functional review of the code in the repository.

2. Types of tests performed

Type Brief description
Manual functional Checking user flows (registration, collection, community, admin, comments).
Basic security Access to protected routes without a token, permissions, body size limits, login/register rate limiting, diagnostic endpoint in production.
Data validation Input normalisation (email, title, status, platform) consistent with the backend.
Automated unit tests Repeatable tests on pure logic (no browser or database), run with Vitest 3.

9. Things I would leave for later (not done yet)

  1. More uniform error messages: right now the error JSON can vary slightly by route. Ideally everything would go through the same place (for example a middleware) so that in production users do not receive long technical details (stack traces, internal information).
  2. Bell without polling every X seconds: today the bell uses polling (asking the server periodically for updates). For a larger product it would be worth considering WebSockets or SSE so the server notifies immediately; that is more deployment and reconnection work if the network drops.
  3. Merge two entries for the same game: if someone has two entries that are really the same title, an admin screen or process would be needed to merge them and move comments and votes to the correct entry (especially if they already share catalogo_id).
  4. Tests that hit the real API: unit tests cover isolated functions; a suite that starts the API (and a test database) and checks whole routes, such as Supertest, would catch integration failures between layers.
