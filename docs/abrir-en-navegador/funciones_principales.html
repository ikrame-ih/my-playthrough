Support syllabus: what spans login and the API

HTTP request and response status codes

Every time the browser (or Thunder Client) calls the API, the server responds with a status number. The ones I use most in this project are:

  • 200 (and 201): everything went well; the response body carries JSON data.
  • 400: the request was malformed or data was missing (validation).
  • 401 Unauthorized: the server cannot identify the user from what was sent in the request. Typical cases: no token, expired token, or signature mismatch. It does not mean “no role permission”; in practice it means invalid or absent identity.
  • 403 Forbidden: the server identifies the user (valid token), but that identity cannot perform the requested action. Example: a normal user on admin routes.
  • 404: resource not found.
  • 500: internal server error (bug or database failure).

Useful rule: 401 = invalid or absent identity; 403 = recognised identity, action not allowed.

JWT (JSON Web Token): what it is and what the payload is

A JWT is a long single string that the server generates when login or registration succeeds. It looks like three chunks separated by dots (xxxxx.yyyyy.zzzzz). It is not the user's password: it is a signed receipt that says, in practice, “whoever shows this is the person with id X, email Y, role Z”, for a limited time.

The middle part (the payload) carries JSON-encoded data (for example id, email, rol). Anyone can read that part by decoding the Base64 (it is not fully encrypted), but cannot forge it without knowing the server secret, because the third part is the signature.

JWT_SECRET (in .env) is the server's “master key” only: with it the token is signed when created and verified when received. That is why documentation calls it a secret: if someone obtains it, they could fabricate fake tokens. It is not committed to Git or sent to the browser.

Session “without storing sessions” on the server

Another common login approach is for the server to store sessions in memory or in a database table (“user 7 connected with session id S”). In this project that table is not needed day to day: the JWT that the client stores (in localStorage) and sends on each request is used. The server only has to verify the signature and expiry: identity on every request without storing an active session list on the server.

Header Authorization: Bearer …

This is the standard format agreed on the web for sending a token with a request. The header is called Authorization; the value starts with the word Bearer, a space, and then the full JWT. Example: Authorization: Bearer eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9…. The authHeaders() function on the client builds that format when a token is stored.

Middleware in Express

In Express, middleware is a function that runs before the route that does the work (for example before listing games). It can read headers, stop with 401, or call next() to continue. authMiddleware acts as a “bouncer”: without a valid JWT you do not pass on private routes.

Why I handle 401 on the client with apiFetch

If the token expired, any protected route can return 401. If I do nothing on the client, the interface would still show the user as logged in even though the server no longer accepts them. That is why I centralised it: if the response is 401, clear token and user from localStorage and reload the page to return to login with a clean state.

1. Authentication: creating the token and verifying it on each request

server/middleware/auth.middleware.js → createToken(user)

What problem it solved

After a successful login or registration, the backend gives the client a portable way to prove identity on subsequent requests, without maintaining a server-side session table for each connection (the JWT model is described in the support syllabus above).

What the function does: uses jwt.sign to generate a JWT whose payload includes id, email and rol, signed with JWT_SECRET and expiring in 7 days. That string is what the browser stores and resends in Authorization: Bearer …. 

jwt.sign(
  { id: user.id, email: user.email, rol: user.rol || "user" },
  JWT_SECRET,
  { expiresIn: "7d" },
);

Note: the JWT payload helps the client with menus and UI, but for sensitive actions (e.g. administration) the server reads the role from the database when needed, because the token can become outdated.

Medium Configure environment variables correctly and do not confuse “data in the token” with “definitive permissions”.

server/middleware/auth.middleware.js → authMiddleware

What problem it solved

Without this middleware, each route would repeat the same code: read Authorization, check the Bearer prefix, extract the token, jwt.verify, and respond with a JSON error if something fails.

What it does: it is the middleware that wraps all of that: if the token is valid, it fills req.user with what came in the payload; if not, it responds with 401 and a clear message.

Low A well-documented Express pattern; the important part is chaining it correctly on the router.

server/middleware/auth.middleware.js → usuarioEsAdmin(usuarioId) and adminMiddleware

What problem it solved

The JWT may have been issued before a role change in the database. It is not enough to read rol only from the payload when certainty about who is an administrator at request time is required.

What it does: usuarioEsAdmin runs SELECT rol FROM usuarios WHERE id = $1. adminMiddleware goes after authMiddleware and, if the DB role is not admin, responds with 403 (here: known identity, action not allowed).

Medium Keep messages clear and do not mix up 401 with 403.

2. Client: headers and a single place for 401

client/src/api.js → authHeaders() and apiFetch(url, opts)

What problem it solved

Without a common layer, each component would repeat how to build JSON and the Bearer header, and the same if (res.status === 401) in multiple places.

What it does: authHeaders returns Content-Type: application/json and, if a token exists in localStorage, the Authorization: Bearer … header. apiFetch wraps fetch and, if the response is 401, clears storage and reloads (behaviour described in the syllabus section on 401). 

if (res.status === 401) {
  localStorage.removeItem("token");
  localStorage.removeItem("user");
  window.location.reload();
  throw new Error("Session expired");
}

Medium A single place to maintain behaviour when the session has expired.

3. Data normalisation before saving

server/utils/normalize.js

What problem it solved

The same address written as Usuario@Gmail.com and as usuario@gmail.com, without normalising, would be treated by the database as two different emails.

What it does: pure functions (no database access) that unify emails, titles and statuses; used in routes and covered by tests in normalize.test.js.

Low–Medium The difficulty is not duplicating rules between client and server.

4. Collection, reusable SQL and covers (CORS)

server/utils/queries.js → queryGamesListForUser(usuarioId)

What problem it solved

A single place with the LEFT JOIN to the catalog avoids repeating the query on each route and centralises official title and image when the entry is linked.

What it does: a SELECT with COALESCE: in plain terms, “if the catalog has a title, use it; otherwise use the user's entry title”. Same for the image URL.

COALESCE(a,b) returns the first non-null value; here it prioritises catalog data without discarding manually entered values.

Medium SQL syntax and handling NULL and spaces in URLs.

server/utils/queries.js → queryGamePublicById(gameId)

What problem it solved

The discussion view shows games from any user; the “only my games” endpoint does not work for that.

What it does: fetches the public entry with owner data (name, etc.) for links and header.

Medium Clearly separate “read someone else's entry in community” from “edit the authenticated user's collection”.

client/src/coverUrl.js → displayCoverUrl(url)

What problem it solved

CORS (Cross-Origin Resource Sharing) is the mechanism by which the browser decides whether a page on one origin (e.g. the Vite frontend) can load resources from another (Steam/RAWG CDN). Sometimes the image does not display if the browser requests it directly from the external domain.

What it does: replaces the URL with a request to our backend /api/covers/proxy?u=…, which downloads the image and serves it from the same origin as the API, avoiding that block in practice.

Low Few lines, but essential for showing covers.

5. Discussion: grouping comments and voting

client/src/components/GameDiscussion.jsx → Map por parent_id

What problem it solved

The server returns a flat list of rows; the interface needs a tree (replies under their parent) and a sensible order for root reviews.

What it does: a Map where each key is a parent_id (or 0 for root) and the value is an array of children; then root reviews are sorted by vote balance and date. That feeds the recursive CommentBlock component.

Note: the Map acts as a key → list index to group children without chaining many manual if statements.

High Visualising the tree and not breaking rendering was the trickiest part.

client/src/components/GameDiscussion.jsx → onVote (useCallback)

What problem it solved

After voting, update only the affected row in React state without reloading the whole page.

What it does: PUT to the vote endpoint and update the comments array with the counters and mi_voto returned by the server. voteBusyId prevents double-click.

Medium

server/routes/games.routes.js → PUT …/comments/:commentId/vote

What problem it solved

A user can only have one vote per comment; votable reviews are only top-level ones (parent_id null).

What it does: validates game and comment; if val === 0 deletes the vote; otherwise, INSERT … ON CONFLICT DO UPDATE to keep one row per (comment, user) pair. Then aggregates votes with SUM and returns mi_voto.

High Test thoroughly with tools like Thunder Client.

6. Recommendation bell (without WebSockets)

client/src/components/NotificationBell.jsx

What problem it solved

Notify about new recommendations without setting up a permanent real-time connection (WebSockets would be a different architecture: a long-lived bidirectional channel). Here I use polling: every so often I ask the server how many unread ones there are.

What it does: ~45 s interval, useRef to compare with the previous count and to know if the user has already interacted (browsers restrict automatic audio), and check whether the tab is visible before playing a sound.

Medium Autoplay audio rules are more troublesome than they seem.
