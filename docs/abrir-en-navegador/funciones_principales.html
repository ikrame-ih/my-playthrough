Repaso: conceptos que uso en casi todo el flujo de login y API

Petición HTTP y códigos de respuesta

Cada vez que el navegador (o Thunder Client) llama a nuestra API, el servidor contesta con un código numérico. Los que más uso en este proyecto son:

  • 200 (y 201): todo fue bien; el cuerpo de la respuesta trae datos JSON.
  • 400: la petición estaba mal formada o faltaban datos (validación).
  • 401 Unauthorized: el servidor no puede identificar al usuario con lo enviado en la petición. Típico: no hay token, el token está caducado o la firma no cuadra. No equivale a “sin permiso de rol”; en la práctica significa identidad no válida o ausente.
  • 403 Forbidden: el servidor identifica al usuario (token válido), pero esa identidad no puede realizar la acción solicitada. Ejemplo: usuario normal en rutas de administración.
  • 404: recurso no encontrado.
  • 500: error interno del servidor (bug o fallo de base de datos).

Recuerdo: 401 = identidad; 403 = identidad clara pero acción prohibida.

JWT (JSON Web Token): qué es y qué es el payload

Un JWT es un texto largo en una sola cadena que el servidor genera al hacer login o registro correcto. Visualmente parece tres trozos separados por puntos (xxxxx.yyyyy.zzzzz). No es una contraseña del usuario: es un recibo firmado que dice “quien presente esto es la persona con id X, email Y, rol Z”, dentro de un tiempo limitado.

La parte central (el payload) lleva datos en JSON codificados (por ejemplo id, email, rol). Cualquiera puede leer esa parte si decodifica el Base64 (no es cifrado total), pero no puede falsificarla sin conocer el secreto del servidor, porque la tercera parte es la firma.

JWT_SECRET (en el .env) es la “clave maestra” solo del servidor: con ella se firma el token al crearlo y se verifica al recibirlo. Por eso en los apuntes se llama secreto: si alguien lo roba, podría fabricar tokens falsos. No se sube a Git ni se envía al navegador.

Sesión “sin guardar sesiones” en el servidor

Otra forma clásica de login es que el servidor guarde en memoria o en base de datos una tabla de sesiones (“usuario 7 conectado con id de sesión S”). Aquí no hace falta esa tabla para el día a día: confiamos en el JWT que el cliente guarda (en localStorage) y envía en cada petición. El servidor solo tiene que verificar la firma y la caducidad. Eso es lo que suelo resumir como: identificar al usuario en cada petición sin almacenar yo misma la lista de sesiones.

Cabecera Authorization: Bearer …

Es el formato estándar acordado en la web para mandar un token junto a la petición. La cabecera se llama Authorization; el valor empieza por la palabra Bearer, un espacio, y después el JWT completo. Ejemplo mental: Authorization: Bearer eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9…. Mi función authHeaders() en el cliente monta exactamente eso cuando hay token guardado.

Middleware en Express

En Express, un middleware es una función intermedia que se ejecuta antes del controlador que responde “de negocio” (por ejemplo antes de listar juegos). Puede leer cabeceras, rechazar la petición con 401, o llamar a next() para dejar pasar al siguiente paso. authMiddleware es, en la práctica, el “portero” que exige JWT válido en rutas privadas.

Por qué trato el 401 en el cliente con apiFetch

Si el token caducó, cualquier ruta protegida puede devolver 401. Si no reacciono, la interfaz seguiría mostrando al usuario como logueado aunque el servidor ya no lo acepte. Por eso centralicé: si la respuesta es 401, borro token y usuario del localStorage y recargo la página para volver a la pantalla de login con un estado limpio.

1. Autenticación: crear el token y comprobarlo en cada petición

server/middleware/auth.middleware.js → createToken(user)

Qué problema resolvía

Después de un login o registro correcto, el backend tiene que entregar al cliente una forma portátil de demostrar quién es en las siguientes peticiones, sin que yo tenga que consultar primero una tabla de “sesiones abiertas” en cada caso (el enfoque con JWT ya lo expliqué arriba).

Qué hace la función: usa jwt.sign para generar un JWT cuyo payload incluye id, email y rol, firmado con JWT_SECRET y con caducidad de 7 días. Ese string es el que el navegador guarda y reenvía en Authorization: Bearer ….

jwt.sign(
  { id: user.id, email: user.email, rol: user.rol || "user" },
  JWT_SECRET,
  { expiresIn: "7d" },
);

Recuerdo: el payload del JWT ayuda al front a mostrar menús, pero para acciones sensibles (como admin) el servidor vuelve a mirar el rol en base de datos cuando hace falta, porque el token podría estar desactualizado.

Media Configurar bien variables de entorno y no confundir “datos en el token” con “permisos definitivos”.

server/middleware/auth.middleware.js → authMiddleware

Qué problema resolvía

Sin esto, en cada ruta tendría que copiar el mismo código: leer la cabecera Authorization, comprobar que empiece por Bearer, extraer el token, llamar a jwt.verify y devolver JSON de error si algo falla.

Qué hace: es el middleware que encapsula todo eso: si el token es válido, rellena req.user con lo que venía en el payload; si no, responde 401 con un mensaje claro.

Baja Patrón muy documentado en Express; lo importante es encadenarlo bien en el router.

server/middleware/auth.middleware.js → usuarioEsAdmin(usuarioId) y adminMiddleware

Qué problema resolvía

El JWT podría haberse emitido antes de que en base de datos cambiara el rol. No basta con leer rol solo del payload si quiero estar segura de quién es administradora ahora mismo.

Qué hace: usuarioEsAdmin ejecuta SELECT rol FROM usuarios WHERE id = $1. adminMiddleware va después de authMiddleware y, si el rol en BD no es admin, responde 403 (aquí sí: identidad conocida, acción no permitida).

Media Mantener claros los mensajes y no mezclar 401 con 403.

2. Cliente: cabeceras y un solo sitio para el 401

client/src/api.js → authHeaders() y apiFetch(url, opts)

Qué problema resolvía

Sin capa común, cada componente repetiría cómo montar JSON y el Bearer, y tendría que copiar el mismo if (res.status === 401) una y otra vez.

Qué hace: authHeaders devuelve Content-Type: application/json y, si existe token en localStorage, la cabecera Authorization: Bearer …. apiFetch envuelve a fetch y, si la respuesta es 401, limpia almacenamiento y recarga (como expliqué en el recuadro del 401).

if (res.status === 401) {
  localStorage.removeItem("token");
  localStorage.removeItem("user");
  window.location.reload();
  throw new Error("Sesión expirada");
}

4. Colección, SQL reutilizable y carátulas (CORS)

server/utils/queries.js → queryGamesListForUser(usuarioId)

Qué problema resolvía

Quería un solo sitio con el LEFT JOIN al catálogo para mostrar título e imagen oficiales cuando la ficha está enlazada, en lugar de repetir la consulta en cada ruta.

Qué hace: un SELECT con COALESCE: en castellano, “si el catálogo tiene título, úsalo; si no, el de la ficha del usuario”. Igual para la URL de imagen.

Recuerdo rápido de COALESCE(a,b): devuelve el primer valor que no sea nulo. Así priorizo datos del catálogo sin perder lo manual.

client/src/coverUrl.js → displayCoverUrl(url)

Qué problema resolvía

CORS (Cross-Origin Resource Sharing) es el mecanismo por el que el navegador decide si una página en un origen (mi React) puede cargar recursos de otro (CDN de Steam/RAWG). A veces la imagen no se muestra si el navegador la pide directamente al dominio externo.

Qué hace: sustituye la URL por una petición a nuestro backend /api/covers/proxy?u=…, que descarga la imagen y la sirve desde el mismo origen que la API, evitando ese bloqueo en la práctica.

Baja Pocas líneas, pero imprescindible para ver carátulas.

5. Discusión: agrupar comentarios y votar

client/src/components/GameDiscussion.jsx → Map por parent_id

Qué problema resolvía

El servidor devuelve una lista plana de filas; en pantalla necesito árbol (respuestas bajo su padre) y un orden razonable para las reseñas raíz.

Qué hace: construyo un Map de JavaScript donde cada clave es un parent_id (o 0 para raíz) y el valor es un array de hijos. Luego ordeno las raíces por saldo de votos y fecha. Eso alimenta el componente recursivo CommentBlock.

Recuerdo: un Map aquí es solo una estructura clave → lista, cómoda para agrupar sin anidar a mano cientos de if.

Alta Visualizar el árbol y no romper el renderizado fue lo más delicado.

client/src/components/GameDiscussion.jsx → onVote (useCallback)
