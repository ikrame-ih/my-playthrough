Temario de apoyo: lo que cruza el login y la API

Petición HTTP y códigos de respuesta

Cada vez que el navegador (o Thunder Client) llama a la API, el servidor contesta con un número de estado. Los que más uso en este proyecto son:

  • 200 (y 201): todo fue bien; el cuerpo de la respuesta trae datos JSON.
  • 400: la petición estaba mal formada o faltaban datos (validación).
  • 401 Unauthorized: el servidor no puede identificar al usuario con lo enviado en la petición. Típico: no hay token, el token está caducado o la firma no cuadra. No equivale a “sin permiso de rol”; en la práctica significa identidad no válida o ausente.
  • 403 Forbidden: el servidor identifica al usuario (token válido), pero esa identidad no puede realizar la acción solicitada. Ejemplo: usuario normal en rutas de administración.
  • 404: recurso no encontrado.
  • 500: error interno del servidor (bug o fallo de base de datos).

Criterio útil: 401 = identidad no válida o ausente; 403 = identidad reconocida, acción no permitida.

JWT (JSON Web Token): qué es y qué es el payload

Un JWT es un texto largo en una sola cadena que el servidor genera cuando el login o el registro salen bien. Se ve como tres trozos separados por puntos (xxxxx.yyyyy.zzzzz). No es la contraseña del usuario: es un recibo firmado que dice, en la práctica, “quien enseñe esto es la persona con id X, email Y, rol Z”, durante un tiempo limitado.

La parte central (el payload) lleva datos en JSON codificados (por ejemplo id, email, rol). Cualquiera puede leer esa parte si decodifica el Base64 (no es cifrado total), pero no puede falsificarla sin conocer el secreto del servidor, porque la tercera parte es la firma.

JWT_SECRET (en el .env) es la “clave maestra” solo del servidor: con ella se firma el token al crearlo y se verifica al recibirlo. Por eso en la documentación se denomina secreto: si alguien lo obtiene, podría fabricar tokens falsos. No se sube a Git ni se envía al navegador.

Sesión “sin guardar sesiones” en el servidor

Otra forma habitual de login es que el servidor guarde en memoria o en base de datos una tabla de sesiones (“usuario 7 conectado con id de sesión S”). En este proyecto no hace falta esa tabla para el día a día: se usa el JWT que el cliente guarda (en localStorage) y envía en cada petición. El servidor solo tiene que verificar la firma y la caducidad: identidad en cada petición sin almacenar en servidor la lista de sesiones activas.

Cabecera Authorization: Bearer …

Es el formato estándar acordado en la web para mandar un token junto a la petición. La cabecera se llama Authorization; el valor empieza por la palabra Bearer, un espacio, y después el JWT completo. Ejemplo: Authorization: Bearer eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9…. La función authHeaders() en el cliente monta ese formato cuando hay token guardado.

Middleware en Express

En Express, un middleware es una función que corre antes de la ruta que hace el trabajo (por ejemplo antes de listar juegos). Puede leer cabeceras, cortar con 401 o llamar a next() para seguir. authMiddleware hace de “portero”: sin JWT válido no pasas en las rutas privadas.

Por qué trato el 401 en el cliente con apiFetch

Si el token caducó, cualquier ruta protegida puede devolver 401. Si no hago nada en el cliente, la interfaz seguiría mostrando al usuario como registrado aunque el servidor ya no lo acepte. Por eso lo centralicé: si la respuesta es 401, borro token y usuario del localStorage y recargo la página para volver al login con estado limpio.

1. Autenticación: crear el token y comprobarlo en cada petición

server/middleware/auth.middleware.js → createToken(user)

Qué problema resolvía

Tras un login o registro correcto, el backend entrega al cliente una forma portátil de demostrar la identidad en las siguientes peticiones, sin mantener una tabla de sesiones en servidor para cada conexión (el modelo JWT está descrito en el bloque de temario anterior).

Qué hace la función: usa jwt.sign para generar un JWT cuyo payload incluye id, email y rol, firmado con JWT_SECRET y con caducidad de 7 días. Ese string es el que el navegador guarda y reenvía en Authorization: Bearer …. 

jwt.sign(
  { id: user.id, email: user.email, rol: user.rol || "user" },
  JWT_SECRET,
  { expiresIn: "7d" },
);

Nota: el payload del JWT sirve al cliente para menús y UI, pero para acciones sensibles (p. ej. administración) el servidor vuelve a leer el rol en base de datos cuando procede, porque el token puede quedar desactualizado.

Media Configurar bien variables de entorno y no confundir “datos en el token” con “permisos definitivos”.

server/middleware/auth.middleware.js → authMiddleware

Qué problema resolvía

Sin este middleware, cada ruta repetiría el mismo código: leer Authorization, comprobar el prefijo Bearer, extraer el token, jwt.verify y responder con JSON de error si falla algo.

Qué hace: es el middleware que encapsula todo eso: si el token es válido, rellena req.user con lo que venía en el payload; si no, responde 401 con un mensaje claro.

Baja Patrón muy documentado en Express; lo importante es encadenarlo bien en el router.

server/middleware/auth.middleware.js → usuarioEsAdmin(usuarioId) y adminMiddleware

Qué problema resolvía

El JWT puede haberse emitido antes de un cambio de rol en base de datos. No basta con leer rol solo del payload si se exige certeza sobre quién es administrador en el momento de la petición.

Qué hace: usuarioEsAdmin ejecuta SELECT rol FROM usuarios WHERE id = $1. adminMiddleware va después de authMiddleware y, si el rol en BD no es admin, responde 403 (aquí sí: identidad conocida, acción no permitida).

Media Mantener claros los mensajes y no mezclar 401 con 403.

2. Cliente: cabeceras y un solo sitio para el 401

client/src/api.js → authHeaders() y apiFetch(url, opts)

Qué problema resolvía

Sin capa común, cada componente repetiría cómo montar JSON y el Bearer, y el mismo if (res.status === 401) en múltiples sitios.

Qué hace: authHeaders devuelve Content-Type: application/json y, si existe token en localStorage, la cabecera Authorization: Bearer …. apiFetch envuelve a fetch y, si la respuesta es 401, limpia almacenamiento y recarga (comportamiento descrito en el apartado de temario sobre el 401). 

if (res.status === 401) {
  localStorage.removeItem("token");
  localStorage.removeItem("user");
  window.location.reload();
  throw new Error("Sesión expirada");
}

3. Normalización de datos antes de guardar

server/utils/normalize.js

Qué problema resolvía

La misma dirección escrita como Usuario@Gmail.com y como usuario@gmail.com, sin normalizar, la base de datos la trataría como dos correos distintos.

Qué hace: funciones puras (sin tocar base de datos) que unifican emails, títulos y estados; se usan en rutas y están cubiertas por tests en normalize.test.js.

4. Colección, SQL reutilizable y carátulas (CORS)

server/utils/queries.js → queryGamesListForUser(usuarioId)

Qué problema resolvía

Un único lugar con el LEFT JOIN al catálogo evita repetir la consulta en cada ruta y centraliza título e imagen oficiales cuando la ficha está enlazada.

Qué hace: un SELECT con COALESCE: en castellano, “si el catálogo tiene título, úsalo; si no, el de la ficha del usuario”. Igual para la URL de imagen.

COALESCE(a,b) devuelve el primer valor que no sea nulo; aquí prioriza datos del catálogo sin descartar lo introducido a mano.

client/src/coverUrl.js → displayCoverUrl(url)

Qué problema resolvía

CORS (Cross-Origin Resource Sharing) es el mecanismo por el que el navegador decide si una página en un origen (p. ej. el frontend en Vite) puede cargar recursos de otro (CDN de Steam/RAWG). A veces la imagen no se muestra si el navegador la pide directamente al dominio externo.

Qué hace: sustituye la URL por una petición a nuestro backend /api/covers/proxy?u=…, que descarga la imagen y la sirve desde el mismo origen que la API, evitando ese bloqueo en la práctica.

Baja Pocas líneas, pero imprescindible para ver carátulas.

5. Discusión: agrupar comentarios y votar

client/src/components/GameDiscussion.jsx → Map por parent_id

Qué problema resolvía

El servidor devuelve una lista plana de filas; la interfaz necesita un árbol (respuestas bajo su padre) y un orden razonable para las reseñas raíz.

Qué hace: un Map donde cada clave es un parent_id (o 0 para raíz) y el valor es un array de hijos; luego se ordenan las raíces por saldo de votos y fecha. Eso alimenta el componente recursivo CommentBlock.

Nota: el Map actúa como índice clave → lista para agrupar hijos sin encadenar muchos if a mano.

Alta Visualizar el árbol y no romper el renderizado fue lo más delicado.

client/src/components/GameDiscussion.jsx → onVote (useCallback)

Qué problema resolvía

Tras votar, actualizar solo la fila afectada en el estado de React sin recargar toda la página.

Qué hace: PUT al endpoint de voto y actualización del array de comentarios con los contadores y mi_voto que devuelve el servidor. voteBusyId evita doble clic.

Media

server/routes/games.routes.js → PUT …/comments/:commentId/vote

Qué problema resolvía

Un usuario solo puede tener un voto por comentario; las reseñas valorables son solo las de primer nivel (parent_id nulo).

Qué hace: valida juego y comentario; si val === 0 borra el voto; si no, INSERT … ON CONFLICT DO UPDATE para mantener una fila por par (comentario, usuario). Luego agrega votos con SUM y devuelve mi_voto.

Alta Probar bien con herramientas tipo Thunder Client.
